Day 42: SSH

Secure Shell & Remote Management

🔐

Telnet vs. SSH

To manage a Cisco router or switch remotely over the network (In-Band management), you must use the VTY (Virtual Teletype) lines. There are two primary protocols for this.

Telnet (TCP Port 23)

The legacy remote access protocol. It is fundamentally insecure.

Critical Flaw: Telnet sends ALL data—including your username, password, and the entire configuration you type—in clear text. Anyone with Wireshark can read it.

SSH (TCP Port 22)

Secure Shell. The modern standard for remote CLI access.

The Fix: SSH uses robust cryptographic algorithms to establish a secure, encrypted tunnel before any data is sent. Eavesdroppers only see gibberish.

📋

The Prerequisites

You cannot just "turn on" SSH on a brand new Cisco device. Because SSH requires encryption, the router must first generate an RSA cryptographic key pair. The router refuses to generate this key until two prerequisites are met.

1. Hostname

The device cannot use the default hostname of "Router" or "Switch". You must change it.

Router(config)# hostname R1

2. Domain Name

You must configure an IP domain name for the device.

R1(config)# ip domain-name cisco.com

Why? The router concatenates the hostname and domain name (e.g., R1.cisco.com) and uses that exact string to name the RSA cryptographic key it generates.

👤 VTY & Local Auth

Unlike a console port which might just ask for a password, SSH requires a Username AND a Password to authenticate the session.

Step 1: The Local Database

You must create a user account stored locally on the router's NVRAM.

username admin secret Cisco123

Step 2: Line VTY Config

You must tell the VTY lines to stop using a generic line password, and instead check the local database you just created.

login local

🪝 Brain Hooks

The Postcard vs Armored Truck

Telnet is a postcard. Anyone handling it (routers/switches in the path) can read your password.
SSH is an armored truck. Even if they intercept the truck, they can't see what's inside.

Transport Input Rule

If you configure SSH, but forget transport input ssh, users can still Telnet in! You must explicitly lock the door.

⌨️ Complete SSH Configuration CLI

1. Fulfill Prerequisites & Create User Router(config)# hostname R1 R1(config)# ip domain-name lab.local R1(config)# username admin privilege 15 secret cisco123
2. Generate Keys & Enforce SSHv2 R1(config)# crypto key generate rsa How many bits in the modulus [512]: 1024 R1(config)# ip ssh version 2 Requires a modulus of at least 768 to enable SSH version 2 (which is vastly more secure than v1).
3. Apply to VTY Lines R1(config)# line vty 0 4 R1(config-line)# login local R1(config-line)# transport input ssh `login local` tells the line to use the username database. `transport input ssh` disables Telnet and forces SSH exclusively.
Verification R1# show ip ssh Confirms SSH is enabled and displays the version (1.99 or 2.0).

🔧 Troubleshooting

"Please define a domain-name first"

Issue: You typed `crypto key generate rsa` and it immediately threw an error.

Fix: You forgot the prerequisites. Ensure you have changed the hostname and set an `ip domain-name`.

"Connection Refused" when connecting

Issue: Keys are generated, but PuTTY/TeraTerm says connection refused on port 22.

Fix: Check the VTY lines. Did you type `transport input ssh`? Did you type `login local` and actually create a username?
