⚡ The "Why" of Configuration Management

Traditional Method

Device-by-device via CLI (SSH/Telnet). Slow, error-prone, hard to scale, lacks version control.

Modern Automation

Centralized controller manages configs as code. Scalable, consistent, easy rollbacks.

Idempotency (Crucial Concept)

Making the same change multiple times yields the same result. The tool only makes changes if the current state differs from the desired state.

A

Ansible

  • ▸ Architecture: Agentless (Connects via SSH/NETCONF). No software installation required on managed devices.
  • ▸ Model: Push. Control node pushes configuration down to nodes.
  • ▸ Core Language: Python (under the hood).
  • ▸ Config Format: YAML. Human-readable data serialization.

Key Terminology

Playbook: The file containing the config instructions (YAML).
Inventory: List of devices to manage.
Module: Scripts that Ansible executes on the device.

P

Puppet

  • ▸ Architecture: Agent-based (typically). Requires a Puppet agent installed on the managed node.
  • ▸ Model: Pull. Agents check in with the Puppet Master (port 8140) to request configs.
  • ▸ Core Language: Ruby.
  • ▸ Config Format: Puppet DSL (Domain Specific Language).

Key Terminology

Puppet Master: The centralized control server.
Manifest: The file containing configuration instructions.
Puppet Agent: Software running on the managed device.

C

Chef

  • ▸ Architecture: Agent-based. Requires Chef client software on managed devices.
  • ▸ Model: Pull. Clients poll the Chef server for updates.
  • ▸ Core Language: Ruby.
  • ▸ Config Format: Ruby (specifically Chef DSL).

Key Terminology

Recipe: A file containing configuration tasks.
Cookbook: A collection of related recipes.
Chef Server/Client: The control plane and managed endpoint.

🎯 Active Recall: Quick Quiz

Test your knowledge. Click on the question to reveal the answer.

1. Which configuration management tool does NOT require installing software on the managed devices?
Ansible. It is "agentless" and connects to devices using standard protocols like SSH or NETCONF.
2. Match the configuration file terms (Playbook, Manifest, Recipe) to their respective tools.
  • Ansible: Playbook
  • Puppet: Manifest
  • Chef: Recipe (grouped in Cookbooks)
3. Which tool utilizes a "Push" model, and what does that mean?
Ansible. In a push model, the control node initiates the connection and pushes the configuration down to the managed devices. (Puppet and Chef use a Pull model where agents poll the server).
4. Ansible playbooks are written in which data serialization format?
YAML (YAML Ain't Markup Language). It relies heavily on indentation for structure.
5. Define "Idempotency" in the context of automation tools.
Idempotency means an operation can be applied multiple times without changing the result beyond the initial application. If a device is already configured correctly, running the playbook/manifest again will make NO changes.
