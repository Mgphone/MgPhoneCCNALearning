Remote Access & SSH

Secure Shell (SSH) & Device Management

Learn how to securely manage network devices remotely. This module covers configuring Local Authentication databases, locking down Console and VTY lines, generating RSA crypto keys, forcing SSH (TCP 22) over Telnet (TCP 23), and applying Access Control Lists to management interfaces.

🧠 Secure Remote Access

Core VTY Security Concepts

  • Telnet vs. SSH Telnet sends all data, including passwords, in plain text. Secure Shell (SSH) establishes an encrypted tunnel using RSA keys before any authentication occurs. Never use Telnet in production.
  • Local Authentication Instead of a single shared password for a line, you can create a local database of users (username [name] secret [pass]) and enforce its use by typing login local on the VTY lines.
  • VTY Line Restrictions VTY (Virtual Teletype) lines handle remote connections. You can force them to reject unencrypted traffic by issuing transport input ssh. You can also attach an ACL using access-class to limit exactly which IP addresses are allowed to connect.

Goal: Secure SW2 with local authentication, enable an SVI for remote management, generate RSA keys, and restrict VTY access via ACL.

Switch> enable

Switch(config)# hostname SW2

! Create local user and enable secret
SW2(config)# enable secret ccna
SW2(config)# username jeremy secret ccna

! Setup Management IP
SW2(config)# interface vlan 1
SW2(config-if)# ip address 192.168.2.253 255.255.255.0
SW2(config-if)# no shutdown
SW2(config-if)# exit
SW2(config)# ip default-gateway 192.168.2.254

! Secure Console Line
SW2(config)# line console 0
SW2(config-line)# login local
SW2(config-line)# exec-timeout 5 0
SW2(config-line)# exit

! Prepare for SSH
SW2(config)# ip domain-name jeremysitlab.com
SW2(config)# crypto key generate rsa
How many bits in the modulus [512]: 2048

! Create an ACL permitting ONLY PC1
SW2(config)# access-list 1 permit host 192.168.1.1

! Secure VTY Lines (Remote Access)
SW2(config)# line vty 0 15
SW2(config-line)# login local
SW2(config-line)# exec-timeout 5 0
SW2(config-line)# transport input ssh
SW2(config-line)# access-class 1 in

Answer: By default, if a Cisco device does not have an enable password/secret configured, it will refuse to let remote users (via Telnet or SSH) enter Privileged EXEC mode. This is a built-in security measure to prevent unauthorized remote users from gaining full control over an unsecured device.

If you try to run crypto key generate rsa on a fresh router, it will fail.

Answer: You must configure a Hostname (other than the default 'Router' or 'Switch') and an IP Domain Name (e.g., ip domain-name cisco.com). The router uses these two strings combined (e.g., SW2.jeremysitlab.com) as the identity to generate the unique mathematical encryption key.

Under the VTY line configuration, use the transport command:

SW2(config)# line vty 0 15
SW2(config-line)# transport input ssh

Once this is entered, any attempt to connect via Telnet (Port 23) will be instantly refused by the device.

Normally, ACLs are applied to physical interfaces using ip access-group. However, for VTY lines, a different command is used.

SW2(config)# line vty 0 15
SW2(config-line)# access-class 1 in

This ensures that even if an attacker has the correct username and password, their connection will be dropped if their IP address is not explicitly permitted in ACL 1.

Command Mode Purpose
username [user] secret [pass] Global Config Creates a local user account with a securely hashed password.

login local Line Config Forces the line to authenticate users against the local username database.
transport input ssh Line Config Restricts remote access exclusively to the secure SSH protocol (blocks Telnet).
