Wireshark & Layer 4 Analysis

Network engineers rely on packet capture software to troubleshoot at a granular level. Move beyond Packet Tracer and learn how to analyze real TCP connections, sequence numbers, flag states, and UDP exchanges using Wireshark.

🧠 Reading TCP/UDP Traffic

TCP vs UDP Highlights

  • TCP Handshakes TCP relies on a 3-Way Handshake (SYN, SYN-ACK, ACK) to establish a connection reliably. To close it, a 4-Way Handshake (FIN, ACK, FIN, ACK) is typically used to ensure both sides agree to terminate.
  • Relative Sequence Numbers Real TCP sequence numbers are randomly generated large numbers (e.g., 1,224,315,781). To make analysis easier, Wireshark translates these into Relative Sequence Numbers starting at 0 for the first packet.
  • Ephemeral Source Ports When your PC browses YouTube (Destination Port 443), it randomly selects a high-numbered Source Port (e.g., 62652) from the Ephemeral Range to uniquely track that specific session's return traffic.

What is SSL in Wireshark?

During a secure web session, after the initial TCP handshake completes, you will see the protocol switch from TCP to SSL (or TLS). This indicates that the TCP payload is now carrying encrypted HTTP data (HTTPS).

Layer 4 Header Size Comparison

Why UDP is faster but less reliable.

TCP requires a minimum of 20 bytes to fit Sequence Numbers, Acknowledgement Numbers, and Control Flags (SYN, ACK, FIN). UDP is a lightweight 8 bytes.

📝 Lab Homework Guide

Because there is no router configuration in this module, your homework is to download Wireshark and perform a live packet capture on your own machine. Check off the steps below as you complete them!

Go to wireshark.org and download the free installer for your OS.

Open Wireshark. You will see a list of network interfaces (e.g., Ethernet, Wi-Fi). Double-click the one that has a jagged line (indicating live traffic) to begin capturing packets. Open a web browser and visit a few websites to generate traffic.

Captures generate thousands of packets a minute. Use the filter bar at the top of Wireshark to narrow it down.

  • Type tcp.port == 443 to show only HTTPS web traffic.
  • Type udp.port == 53 to show only DNS queries/responses.
  • Type tcp.flags.syn == 1 to easily locate the start of handshakes.

Press Enter after typing to apply the filter. The bar will turn green if the syntax is valid.

Look for three consecutive TCP packets between the same two IP addresses.

  1. SYN: PC sends to Server. Seq=0. (Synchronize bit set).
  2. SYN, ACK: Server replies to PC. Seq=0, Ack=1.
  3. ACK: PC replies to Server. Seq=1, Ack=1.

Expand the "Transmission Control Protocol" section in the bottom pane to see the Flags and Sequence numbers.

Scroll down your capture and look for packets with the FIN flag set. This indicates one side wishes to close the connection.

A clean teardown is a 4-way process:

  1. FIN, ACK: Device A wants to close.
  2. ACK: Device B acknowledges Device A's closure.
  3. FIN, ACK: Device B wants to close its side.
  4. ACK: Device A acknowledges Device B's closure.

Note: Real-world captures can be messy. You might see FIN/ACK combined, or RST (Reset) packets abruptly killing connections. This is normal.
