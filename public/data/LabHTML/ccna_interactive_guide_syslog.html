Syslog Operations

System Message Logging (Syslog)

Learn how Cisco routers generate, format, and store system event logs. We will explore the 8 severity levels, how to route logs to different destinations (Console, VTY, Buffer, External Servers), and how to apply filters using the logging trap command.

🧠 Core Syslog Mechanics

Logging Destinations

  • Console Line (Default) Logs are automatically printed directly to the console port. If you are physically connected via console, you will see events in real-time.
  • VTY Lines (Telnet/SSH) Logs are NOT displayed by default when connected remotely. You must issue the terminal monitor command to view real-time logs in your session.
  • Internal Buffer Logs can be saved to the router's RAM (Buffer). You can view them anytime using the show logging command. The buffer clears if the router reboots.
  • External Syslog Server A dedicated external server (NMS) that receives and stores logs centrally. This is crucial for long-term historical analysis and persistent storage.

The "Trap" Filtering Rule

When you set a logging level (e.g., logging trap warnings), it operates as a ceiling. The router will send logs matching that exact level, AND all levels with a lower severity number (higher urgency).

Example: Level 4 (Warnings) includes Levels 0, 1, 2, 3, and 4. It drops Levels 5, 6, and 7.

Syslog Severity Levels (0-7)

Lower numbers indicate higher urgency.

Mnemonic: Every Action Critical, Every Warning Noticeable, Information Detailed.

💻 Device Configuration Lab

Review the commands to enable timestamps, log to the internal buffer, monitor VTY sessions, and route logs to external servers.

Goal: Add accurate timestamps to log messages and enable saving logs to the router's internal RAM buffer.

R1> enable
R1# configure terminal

! Add timestamps (with milliseconds) to all syslog messages
R1(config)# service timestamps log datetime msec

! Enable logging to the internal RAM buffer
R1(config)# logging buffered

! (Optional) Set the specific size of the buffer in bytes
R1(config)# logging buffered 8192

! Verify the buffer status and view stored logs
R1(config)# do show logging
Syslog logging: enabled
    Console logging: level debugging, 18 messages logged
    Buffer logging: level debugging, 18 messages logged
...

Goal: Point the router to an external Syslog Server and restrict the volume of messages sent using a trap filter.

R1> enable
R1# configure terminal

! Specify the IP address of the external Syslog Server
! ('logging' and 'logging host' perform the exact same function)
R1(config)# logging host 192.168.1.100

! Filter out "chatter" by only sending events that are Warnings (Level 4) or more severe
R1(config)# logging trap warnings

! Verify external logging setup
R1(config)# do show logging
...
    Trap logging: level warnings, 15 message lines logged
    Logging to 192.168.1.100 (udp port 514)

Goal: Connect to the router remotely via Telnet/SSH and enable real-time logs to display on the remote terminal screen.

! From PC1's Command Prompt
C:\> telnet 192.168.1.1
User Access Verification
Password:
R1> enable
Password:

! By default, if an interface goes down right now, we won't see the alert!
! We MUST enable terminal monitoring for this specific session.
R1# terminal monitor

! Now, events will print directly to our remote screen in real-time.
R1# configure terminal
R1(config)# interface g0/1
R1(config-if)# shutdown
*Mar 1 01:12:44.205: %LINK-5-CHANGED: Interface GigabitEthernet0/1, changed state to administratively down
Note: The terminal monitor command only applies to the current remote session. If you close Telnet/SSH and reconnect, you must type the command again.

📝 Lab Tasks & Solutions

Check off the steps as you complete them to save your progress.

When you type shutdown, you see messages for both %LINK-5-CHANGED and %LINEPROTO-5-UPDOWN.

Answer: The first message corresponds to the Layer 1 physical status (Status column in show ip int brief). The second message corresponds to the Layer 2 data link protocol status (Protocol column). Since L2 relies on L1, bringing down the interface triggers an alert for both layers simultaneously.

By default, logs just show the message. Adding timestamps is critical for correlating events during troubleshooting.

R1(config)# service timestamps log datetime msec

*In Packet Tracer, the msec (milliseconds) keyword is mandatory for the command to be accepted, though it is optional in real IOS.

Log into R1 from PC1. Once inside Privileged EXEC mode, issue the monitor command.

R1# terminal monitor

If you don't run this, your remote terminal screen will be completely silent regarding real-time network events.

! Enable Buffer Logging (default size is usually fine, or specify bytes)
R1(config)# logging buffered 8192

! Point to the External Syslog Server
R1(config)# logging host 192.168.1.100

! Optional: Set trap severity (PT only supports 'debugging')
R1(config)# logging trap debugging

📋 Command Reference

🔍
Command Mode Purpose
service timestamps log datetime msec Global Config Appends accurate timestamps (down to the millisecond) to every generated log message.
terminal monitor Privileged EXEC Enables real-time console log readouts over a remote VTY session (Telnet/SSH).
logging buffered [bytes] Global Config Saves log messages into the router's internal RAM so they can be viewed later.
logging [host|ip] Global Config Configures the router to send logs to an external centralized Syslog Server.
