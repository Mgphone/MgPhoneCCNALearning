DHCP Exhaustion Attack

Discover how vulnerabilities in the DORA process allow an attacker to starve a network of IP addresses. In this demo, we use Kali Linux and Yersinia to execute a Denial of Service (DoS) attack against a Cisco router acting as a DHCP server.

The Vulnerability: DORA & MAC Spoofing

The DHCP protocol lacks built-in authentication. When an attacker floods the network with DHCP DISCOVER messages, the server faithfully replies with offers, temporarily reserving those IPs.

  • Why spoof MAC Addresses? If the attacker used their real MAC address repeatedly, the DHCP server would just offer the same IP address over and over. By spoofing thousands of random MACs, the attacker tricks the server into thinking thousands of unique hosts are joining.
  • The Consequence (DoS & APIPA) Once the pool reaches 100% capacity, legitimate users cannot get an IP address. Windows PCs will give up and self-assign an APIPA address in the 169.254.X.X range, breaking network connectivity.

Terminal Demo Commands

Review the commands used by Jeremy during the Kali Linux and Cisco Router demonstration.

Check the status of the pool before, during, and after the attack.

! View the configured pool parameters
R1# show run | section dhcp

! View total leased vs available addresses
R1# show ip dhcp pool
Pool POOL1 :
Utilization mark (high/low) : 100 / 0
Total addresses : 254
Leased addresses : 253

! View the specific MAC to IP address mappings (The bindings)
! During an attack, this table will be massively filled with random MACs
R1# show ip dhcp binding

! TO RECOVER: Clear all leased bindings from the router's memory
R1# clear ip dhcp binding *

Launch the Yersinia network tool to execute the starvation attack.

# Open the Yersinia Graphical User Interface (GUI)
kali@kali:~$ yersinia -G

# 1. Click "Launch Attack"
# 2. Select "DHCP"
# 3. Choose "sending DISCOVER packet" (Causes DoS)
# 4. Click OK

Observe the behavior of a Windows client failing to find a DHCP server.

C:\> After admin clears router bindings, force the PC to request a new IP

📝 Lab Demonstration Flow

Check off the steps to track the chronological flow of Jeremy's demo.

Verify that R1 has a configured pool (e.g., 192.168.1.0/24) and that show ip dhcp pool indicates 0 leased addresses.

Open Yersinia (yersinia -G) and launch the DHCP "DISCOVER packet" DoS attack. Immediately, checking R1's bindings will show 253 leased addresses assigned to random spoofed MACs.

When connecting PC1, it fails to receive an IP. Using ipconfig /all reveals an autoconfigured IPv4 address starting with 169.254.X.X. Pings to the default gateway (R1) fail.

Disconnect Kali Linux, run clear ip dhcp binding * on R1 to empty the pool, and then run ipconfig /renew on PC1 to successfully grab an IP address.
