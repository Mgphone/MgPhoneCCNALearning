Voice VLANs & VLAN Leaking

Discover how Cisco IP Phones act as mini-switches, allowing a PC and a Phone to share a single physical switchport while keeping their traffic in separate, isolated VLANs. Additionally, explore the severe security vulnerability known as VLAN Leaking.

🧠 IP Phones & 802.1Q Tagging

The 3-Port Mini Switch

A Cisco IP Phone contains a built-in 3-port switch to save wiring costs (you only need one wall jack for a desk):

  • Port 1
    Connects to the upstream Cisco Switch.
  • Port 2
    Connects to the downstream PC.
  • Port 3
    Internal port connecting to the phone's voice hardware.

Tagging on an Access Port

When you configure switchport voice vlan [id] on an access port, the switch uses CDP (Cisco Discovery Protocol) to instruct the phone to attach an 802.1Q tag to its voice traffic.

The PC's data traffic passes through the phone untagged. The switch places the untagged traffic into the Access VLAN, and the tagged traffic into the Voice VLAN.

Frame Structure Comparison

How the switchport interprets incoming traffic.

Hex 0x8100 in the EtherType field indicates the presence of an 802.1Q VLAN tag.

⚠️ Security Threat: VLAN Leaking

If two connected switches have mismatched Native VLANs on a trunk (e.g., SW1 uses 20, SW2 uses 10), untagged traffic from VLAN 20 on SW1 will cross the trunk and be deposited into VLAN 10 on SW2. This allows devices in different subnets/VLANs to bypass the router, a major security flaw.

💻 Device Configuration Lab

Review the commands to configure Voice VLANs, ROAS for multiple subnets, and Native VLAN trunks.

Goal: Configure the switchports connected to the phones to handle both Data (VLAN 10) and Voice (VLAN 20) traffic simultaneously.

SW1> enable
SW1# configure terminal

! Apply configuration to the phone interfaces
SW1(config)# interface range g1/0/2 - 3

! Assign the untagged Data VLAN for the PC
SW1(config-if-range)# switchport mode access
SW1(config-if-range)# switchport access vlan 10

! Assign the tagged Voice VLAN for the IP Phone
! (Uses CDP to tell the phone to tag its frames with 802.1Q ID 20)
SW1(config-if-range)# switchport voice vlan 20

SW1(config-if-range)# exit

! Configure the trunk uplink to the router, allowing both VLANs
SW1(config)# interface g1/0/1
SW1(config-if)# switchport mode trunk
SW1(config-if)# switchport trunk encapsulation dot1q
SW1(config-if)# switchport trunk allowed vlan 10,20

Goal: Configure Router-on-a-Stick to act as the default gateway for both the Data PC subnet and the Voice Phone subnet.

R1> enable
R1# configure terminal

! Enable physical interface
R1(config)# interface f0/0
R1(config-if)# no shutdown
R1(config-if)# exit

! Configure Data Subinterface (VLAN 10)
R1(config)# interface f0/0.10
R1(config-subif)# encapsulation dot1q 10
R1(config-subif)# ip address 192.168.10.1 255.255.255.0
R1(config-subif)# exit

! Configure Voice Subinterface (VLAN 20)
R1(config)# interface f0/0.20
R1(config-subif)# encapsulation dot1q 20
R1(config-subif)# ip address 192.168.20.1 255.255.255.0

Goal: From the Boson Demo. Deliberately misconfigure the Native VLAN on a trunk link to observe VLAN Leaking.

! PC2 is in VLAN 20. PC3 is in VLAN 10. They share the 10.0.0.x/24 subnet.
! Normally they cannot ping each other without a router.

! On Switch 1 (Trunk Interface)
SW1(config)# interface f0/4
SW1(config-if)# switchport trunk native vlan 20
! CDP Warning: Native VLAN mismatch discovered on FastEthernet0/4 (20), with Switch2 FastEthernet0/1 (10).

! On Switch 2 (Trunk Interface)
SW2(config)# interface f0/1
SW2(config-if)# switchport trunk native vlan 10

! The Leak Result:
1. PC2 sends untagged frame to SW1. SW1 puts it in Access VLAN 20.
2. SW1 forwards it out trunk. Because Native VLAN = 20, SW1 sends it UNTAGGED.
3. SW2 receives untagged frame. Because its Native VLAN = 10, it places it in VLAN 10.
4. SW2 forwards it to PC3. The Ping succeeds! The traffic "leaked" across broadcast domains.

📝 Lab Tasks & Solutions

Check off the steps as you complete them to save your progress.

We configure the port as an access port for data, and then apply the voice vlan command.

SW1(config)# interface range g1/0/2 - 3
SW1(config-if-range)# switchport mode access
SW1(config-if-range)# switchport access vlan 10
SW1(config-if-range)# switchport voice vlan 20

Answer:

  • When checking the outbound PDU of the PC1 Ping arriving at the switch, there is NO 802.1Q header. The switch receives it untagged and processes it as VLAN 10 traffic.
  • When checking the outbound PDU of the Phone Call arriving at the switch, there IS an 802.1Q header. The TCI field shows 0014 (Hex for 20). The switch receives it tagged and processes it as VLAN 20 traffic.

This is a security vulnerability where traffic bleeds between isolated domains.

! On SW1:
SW1(config-if)# switchport trunk native vlan 20

! On SW2:
SW2(config-if)# switchport trunk native vlan 10

CDP will immediately generate console warnings complaining about a Native VLAN mismatch.

Answer: YES.

Assuming they share the same IP subnet (10.0.0.0/24), the ping works. SW1 receives PC2's frame and sends it across the trunk. Since it belongs to SW1's Native VLAN (20), SW1 sends it completely untagged. SW2 receives an untagged frame on a trunk. SW2 assumes untagged frames belong to ITS Native VLAN (10). SW2 forwards the frame to PC3. Traffic successfully leaked across the broadcast domains!
