ARP & MAC Tables

Analyzing Ethernet Switching

This module covers how switches populate their MAC address tables dynamically and how end devices (PCs) use the Address Resolution Protocol (ARP) to map IP addresses to MAC addresses on a local network.

The Bridging Gap

Devices use IP addresses (Layer 3) to identify who they want to talk to, but they must use MAC addresses (Layer 2) to build the Ethernet frame and deliver the data on the local segment.

🔍

The ARP Solution

When a PC knows a destination IP but not the MAC address, it broadcasts an ARP Request (FFFF.FFFF.FFFF) asking, "Who has this IP?". The owner of that IP responds with a unicast ARP Reply.

📥

Switch Learning (Source MAC)

Switches start with empty tables. Every time a frame enters a port, the switch records the Source MAC Address against that incoming port.

📤

Switch Forwarding (Dest MAC)

The switch checks the Destination MAC Address. If known, it forwards out the specific port. If unknown (or broadcast), it floods the frame out all other ports.

Frame Types in Initial Connection

A single ping to an unknown destination actually generates four distinct frames initially.

The first frame is always a broadcast (ARP Req), impacting all local network bandwidth, while subsequent frames are unicast.

Because tables are empty, PC1 cannot immediately build an Ethernet frame. The following sequence occurs:

  1. ARP Request (Broadcast): PC1 sends an ARP request asking "Who has PC3's IP?". The destination MAC is FFFF.FFFF.FFFF.
    Receivers: Both switches receive and flood this frame out all active ports. PC2, PC3, and PC4 receive it. PC2 and PC4 drop it; PC3 processes it.
  2. ARP Reply (Unicast): PC3 replies directly to PC1 with its MAC address.
    Receivers: SW2 receives it, learns PC3's MAC, and forwards to SW1. SW1 receives it, learns PC3's MAC, and forwards directly to PC1.
  3. ICMP Echo Request (Unicast): Now possessing the MAC, PC1 sends the actual Ping packet.
    Receivers: SW1 forwards to SW2, SW2 forwards to PC3.
  4. ICMP Echo Reply (Unicast): PC3 replies to the ping.
    Receivers: SW2 forwards to SW1, SW1 forwards to PC1.

To execute this in Packet Tracer:

  • Click the Simulation tab (bottom right corner).
  • Open PC1's Desktop, then open the Command Prompt.
  • Run the command: ping 192.168.1.3
  • Click the Play / Capture Forward button in the Simulation Panel to advance the animation frame by frame.
  • You will visually see the ARP broadcast hit all devices, followed by the unicast paths of the ARP reply and ICMP packets.

Switches learn Source MAC addresses dynamically when a device sends a frame. To populate the tables completely, every PC must send traffic across the network.

! From PC1's Command Prompt:
PC1> ping 192.168.1.2 # Populates PC1 & PC2
PC1> ping 192.168.1.3 # Populates PC3

! From PC2's Command Prompt:
PC2> ping 192.168.1.4 # Populates PC4

*Once traffic traverses all switch ports across both switches, their MAC tables are fully populated with all PC addresses.

Go to the CLI of each switch and enter Privileged EXEC mode.

SW1> enable
SW1# show mac address-table
Topology Note: On SW1, you will clearly see PC1 on Fa0/1 and PC2 on Fa0/2. However, PC3 and PC4 will both appear on the uplink port (Gig0/1). To find their specific edge ports, you must run the same command on SW2.

To reset the lab state and clear the switch's learned MAC addresses, execute the following command in Privileged EXEC mode on both SW1 and SW2:

SW1# clear mac address-table dynamic
SW2# clear mac address-table dynamic

*Note: Packet Tracer does not support clearing specific interfaces (e.g., clear mac address-table dynamic interface Fa0/1). You must clear the entire dynamic table.
