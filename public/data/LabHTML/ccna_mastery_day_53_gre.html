GRE Tunnels (Generic Routing Encapsulation)

Connect remote private networks securely across the public Internet. GRE allows you to create virtual point-to-point links (Overlay) by encapsulating private packets inside public IPv4 headers (Underlay).

Underlay vs. Overlay Networks

GRE encapsulates the entire original packet inside a new header, creating a logical network on top of a physical one.

  • The Underlay (Physical) The public internet or physical infrastructure. The tunnel source and tunnel destination use public IP addresses to route packets across the internet.
  • The Overlay (Logical) The virtual tunnel itself. The ip address assigned to the tunnel interface is a private IP, making the two remote routers act like they are directly connected.
  • Overhead (24 Bytes) GRE encapsulation adds 24 bytes of overhead to every packet (20 Bytes for the new IPv4 Header + 4 Bytes for the GRE Header).

Packet Size: Payload vs Overhead

Because GRE adds 24 bytes, the MTU (Maximum Transmission Unit) drops from the standard 1500 to 1476.

Device Configurations

Goal: Create the logical tunnel interface on Router 1 and map it to the physical internet link.

! Create the virtual tunnel interface
R1(config)# interface tunnel 0

! Assign a private IP address to the tunnel (The Overlay Network)
R1(config-if)# ip address 10.0.0.1 255.255.255.252

! Define the Underlay parameters (The Public Internet Path)
! Source is R1's own physical internet interface or public IP
R1(config-if)# tunnel source g0/0

! Destination is R2's public IP address
R1(config-if)# tunnel destination 2.2.2.2

Goal: Create the matching tunnel interface on Router 2. The source/destination must be mirrored.

! Create the matching virtual tunnel interface
R2(config)# interface tunnel 0

! Assign a private IP address in the same /30 subnet as R1
R2(config-if)# ip address 10.0.0.2 255.255.255.252

! Mirror the Underlay parameters
R2(config-if)# tunnel source g0/0
R2(config-if)# tunnel destination 1.1.1.1

Goal: Route traffic over the new tunnel. Because GRE supports multicast, we can use OSPF!

! On R1: Advertise the internal LAN and the Tunnel network into OSPF
R1(config)# router ospf 1
R1(config-router)# network 192.168.1.0 0.0.0.255 area 0
R1(config-router)# network 10.0.0.0 0.0.0.3 area 0

! On R2: Advertise the internal LAN and the Tunnel network
R2(config)# router ospf 1
R2(config-router)# network 192.168.2.0 0.0.0.255 area 0
R2(config-router)# network 10.0.0.0 0.0.0.3 area 0

! Wait a few seconds... OSPF Adjacency will form OVER the internet!

📝 Lab Walkthrough Checklist

Check off the steps as you complete them to save your progress.

Before building the tunnel, R1 must be able to ping R2's public IP address over the physical internet connection. If ping fails, the tunnel cannot be built.

Use interface tunnel 0 and assign a private IP address (e.g., 10.0.0.1 255.255.255.252) which acts as the direct link between the two sites.

Configure tunnel source (usually the local outbound interface) and tunnel destination (the remote router's public IP address).

Use static routes pointing to the tunnel interface, or enable a routing protocol like OSPF on the tunnel interface to dynamically advertise the remote LANs.
