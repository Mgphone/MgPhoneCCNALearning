Troubleshooting Static Routes

Put your troubleshooting skills to the test! PC1 cannot ping PC2. In this lab, we dive into the CLI to hunt down configuration errors in the routing tables and interfaces of R1, R2, and R3. One misconfiguration hides on each router.

Static Route Mechanics & Overwrites

Understanding how Cisco IOS handles modifications to routing configurations is critical for troubleshooting.

  • Overwriting Interfaces (Replaces) If you configure an IP address on an interface that already has one (e.g., ip address 192.168.23.3...), the router automatically overwrites the old IP. You do not need to delete it first.
  • Overwriting Routes (Appends) If you configure a new static route to a destination network that already has one, the router adds both routes to the routing table and load-balances across them. To fix a bad route, you MUST explicitly delete the wrong one using no ip route....
  • Verifying PC Settings Always check layer 3 fundamentals on end hosts using ipconfig /all. Ensure the IP, subnet mask, and default gateway align with the network topology.

Administrative Distance (Trustworthiness)

Static routes are highly trusted by the router. Lower AD = More trusted.

Device Configurations

Issue: R1 has a static route to the 192.168.3.0/24 network, but the next-hop IP is configured as 192.168.12.3 (doesn't exist). It should point to R2's interface at 192.168.12.2.

! Find the incorrect route in the running config
R1# show running-config | include ip route
ip route 192.168.3.0 255.255.255.0 192.168.12.3

R1# configure terminal
! Step 1: Delete the incorrect route
R1(config)# no ip route 192.168.3.0 255.255.255.0 192.168.12.3

! Step 2: Add the correct route pointing to R2
R1(config)# ip route 192.168.3.0 255.255.255.0 192.168.12.2

Issue: R2's static route to 192.168.3.0/24 is trying to exit out of g0/0 (the interface facing R1), but R3 is connected out of g0/1.

! View the routing table to see the bad exit interface
R2# show ip route
S 192.168.3.0/24 is directly connected, GigabitEthernet0/0

R2# configure terminal
! Step 1: Delete the incorrect route (Crucial, or else R2 will load-balance!)
R2(config)# no ip route 192.168.3.0 255.255.255.0 g0/0

! Step 2: Add the correct route exiting G0/1 towards R3
R2(config)# ip route 192.168.3.0 255.255.255.0 g0/1

Issue: R3's G0/0 interface (facing R2) was accidentally configured with the IP address 192.168.13.3 instead of the correct subnet address 192.168.23.3.

! Check interface IPs
R3# show ip interface brief
GigabitEthernet0/0 192.168.13.3 YES manual up up

R3# configure terminal
R3(config)# interface g0/0

! Fix the IP address (No need to use 'no', new IP overwrites the old one)
R3(config-if)# ip address 192.168.23.3 255.255.255.0

📝 Lab Walkthrough Checklist

Check off the steps as you complete them to save your progress.

Go to PC1's command prompt and try to ping 192.168.3.1. It fails. Verify PC1's configuration with ipconfig /all. Ensure it can ping its default gateway at 192.168.1.254 (R1). It can, so the issue lies beyond the local LAN.

On R1, check show ip route. Notice the static route points to 192.168.12.3. Negate the wrong route with no ip route..., then add the correct one pointing to 192.168.12.2.

On R2, check the routing table. The route to 192.168.3.0 goes out G0/0 instead of G0/1. Remove the bad route and apply the new one pointing out G0/1. Remember: if you don't delete the bad route, R2 will keep both and load balance!

On R3, use show ip interface brief. Notice G0/0 has an IP in the wrong subnet (13.x instead of 23.x). Enter interface config mode and assign ip address 192.168.23.3 255.255.255.0 to overwrite it.
