DHCP Exhaustion Attack (Kali Linux Demo)

Understanding network vulnerabilities: How an attacker starves a DHCP server and causes a Denial of Service (DoS).

1. Executive Summary

This lab provides a hands-on demonstration of a DHCP Exhaustion (Starvation) attack using the Yersinia tool in Kali Linux. By flooding a DHCP server with fake DHCP Discover messages paired with spoofed MAC addresses, an attacker can rapidly consume all available IP addresses in the DHCP pool. This results in a Denial of Service (DoS), preventing legitimate users from acquiring IP addresses and accessing the network.

2. Conceptual Breakdown

The Vulnerability (DORA)

Normally, DHCP follows four steps: Discover, Offer, Request, Acknowledge. When the server receives a Discover message, it implicitly trusts the client and temporarily reserves an IP address when sending the Offer. The protocol lacks built-in authentication for these requests.

MAC Spoofing

If an attacker repeatedly sent requests from the same MAC address, the server would just offer the same IP. To circumvent this, the attacker continually spoofs (fakes) random source MAC addresses, tricking the server into believing thousands of unique hosts are joining the network.

The Consequence: APIPA Fallback

Once the DHCP pool is completely exhausted, legitimate hosts are denied IP addresses. On Windows machines, failure to reach a DHCP server triggers a fallback mechanism called APIPA (Automatic Private IP Addressing), auto-assigning an address in the 169.254.x.x range. This prevents communication beyond the local link.

3. Step-by-Step Attack & Recovery Flow

Step 1: Verify Initial DHCP Status (Pre-Attack)

Before the attack, verify that the router has available IP addresses and an empty binding table.

R1# show ip dhcp pool
R1# show ip dhcp binding

Step 2: Launch the DHCP Starvation Attack

On the Kali Linux attacker machine, open the Yersinia tool in GUI mode to initiate the flooding of DISCOVER packets.

kali@kali:~$ yersinia -G
# (In GUI: Select DHCP -> Launch Attack -> "sending DISCOVER packet")

Step 3: Observe the Damage

Return to the router. The pool will now show all available addresses as leased.

R1# show ip dhcp binding
Exam Note: When checking the binding table during an attack, you will see a massive list of IP addresses leased out to random, spoofed MAC addresses.

Step 4: Legitimate Client Failure & Recovery

PC1 fails to get an IP and defaults to an APIPA address. To fix this, disconnect the attacker, clear the poisoned bindings on the router, and force the PC to renew its IP.

C:\> ipconfig /all # PC1 shows 169.254.x.x
R1# clear ip dhcp binding * # Clears the poisoned table
C:\> ipconfig /renew # PC1 successfully grabs an IP

4. Command Reference Table

Device Command Mode Description
Router show run | section dhcp Privileged EXEC Displays only the DHCP pool configuration from the running config.
Router show ip dhcp pool Privileged EXEC Shows DHCP pool statistics, including total and currently leased addresses.
Router show ip dhcp binding Privileged EXEC Displays a table mapping leased IP addresses to their corresponding MAC addresses.
Router clear ip dhcp binding * Privileged EXEC Clears all dynamically assigned IP addresses from the binding table.
Kali Linux yersinia -G Terminal Launches the graphical user interface (GUI) for the Yersinia attack tool.
Windows PC ipconfig /renew Command Prompt Forces the PC to contact the DHCP server to request or renew an IP address.

5. Verification & Troubleshooting

  • Spotting the Attack on the Network Run show ip dhcp binding. If the table is suddenly saturated with entries mapped to random, chaotic MAC addresses, an attacker is actively exhausting your pool.
  • Client Diagnosis If a user complains of "no internet" and their ipconfig output displays an IP address starting with 169.254.X.X, their computer gave up waiting for a DHCP response and self-assigned an APIPA address.

🧠 Knowledge Check Flashcards

Question: What specific DHCP message does an attacker flood the server with during a Starvation attack?
Answer: The attacker floods the server with DHCP DISCOVER messages.
Question: Why must the attacker spoof the source MAC address during a DHCP Exhaustion attack?
Answer: If the MAC address remained static, the DHCP server would recognize it as the same host and repeatedly offer the exact same single IP address. Spoofing tricks the server into thinking thousands of unique, new hosts are requesting IPs.
Question: What IPv4 address range is automatically assigned to a Windows PC if it fails to reach a DHCP server?
Answer: It falls back to the APIPA (Automatic Private IP Addressing) range, which is 169.254.0.0/16.
Question: What Cisco IOS command is used to remove all dynamically assigned IP addresses from the DHCP binding table?
Answer: clear ip dhcp binding *
